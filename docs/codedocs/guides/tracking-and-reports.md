---
title: "Enable Tracking And Reports"
description: "Turn a local assessment run into a tracked and reportable workflow with MLflow and PDF output."
---

RAITAP already writes local artifacts by default. This guide adds two operational features on top of that baseline: MLflow tracking and PDF report generation.

## Problem

Local files are useful for debugging, but teams usually need searchable run metadata and a shareable report artifact.

## Solution

Enable the MLflow tracking preset and keep the reporting preset active. RAITAP will write artifacts locally first, then forward them to MLflow.

<Steps>
<Step>
### Install the required extras
```bash
uv add "raitap[captum,metrics,mlflow,reporting,torch-cpu]"
```
</Step>
<Step>
### Run with tracking enabled
```bash
uv run raitap \
  hardware=cpu \
  model=resnet50 \
  data=imagenet_samples \
  transparency=demo \
  tracking=mlflow \
  reporting=pdf
```
</Step>
<Step>
### Inspect the local report and the MLflow UI
```bash
find outputs -maxdepth 4 -type f | sort
```
</Step>
</Steps>

Relevant config block from `src/raitap/configs/tracking/mlflow.yaml`:

```yaml
_target_: MLFlowTracker
output_forwarding_url: http://127.0.0.1:5000
log_model: false
open_when_done: true
```

What RAITAP logs:

- config JSON via `MLFlowTracker.log_config()`
- dataset metadata via `Data.describe()`
- scalar metrics via `MetricsEvaluation.log()`
- transparency artifacts via `ExplanationResult.log()` and `VisualisationResult.log()`
- report artifacts via `ReportGeneration.log()`

If the tracking URI points to localhost and nothing is listening on the configured port, `MLFlowTracker._ensure_server_running()` in `src/raitap/tracking/mlflow_tracker.py` attempts to start a local MLflow server automatically.

The PDF itself is generated by `PDFReporter.generate()` in `src/raitap/reporting/pdf_reporter.py`. The report builder stages images under `reports/_assets`, writes `report.pdf`, and also writes `report_manifest.json` so merged reports and external tooling can inspect the report structure without parsing the PDF.

If you want a report but no tracking, set `tracking._target_=null`. If you want tracking but no report, set `reporting=disabled`. The two features are independent, but they work well together because both consume the same artifact-owning result objects.

For teams comparing multiple runs, this is the practical turning point where RAITAP starts behaving like an assessment service instead of a local script. The local run directory remains the source of truth, while MLflow and the generated PDF become searchable and shareable views over the same artifacts.
